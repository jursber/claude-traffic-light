# 串口被占用（拒绝访问）— 用户排查指南

在 Windows 上打开 COM 口时出现 **`PermissionError(13, 拒绝访问)`**，几乎总是表示：**该串口已被其它进程以独占方式打开**，本程序无法再打开第二次。

可先尝试 **拔掉 USB 再插上**（让占用方释放句柄），再在本程序里点 **刷新 → 连接**。若仍失败，按下文用任务管理器或 Process Explorer 精查。

---

## 1. 最常见是谁占着口？

| 可能占用者 | 说明 |
|------------|------|
| **灯控后台**（`daemon_unified.py`） | 正常工作时独占 USB 串口/BLE，向灯发指令。试灯点「连接」前，本程序会尝试结束该进程。 |
| **Arduino IDE「串口监视器」** | 打开监视器即占用 COM，需先关掉监视器窗口。 |
| **串口调试助手 / SSCOM / MobaXterm 等** | 任何连上该 COM 的工具都要退出或断开。 |
| **另一个本程序实例** | 若误开两个配置台且都点了「连接」，也会抢同一口。 |
| **其它 IDE 串口插件** | 例如某些嵌入式插件在后台打开串口。 |

---

## 2. 推荐操作顺序（用户版）

1. 关掉 **Arduino 串口监视器**、各种 **串口助手**。  
2. **拔插 USB** 或重启占用串口的软件，再在本程序点 **刷新**。  
3. 打开 **任务管理器 → 详细信息**，查找命令行里含 **`daemon_unified`** 的 `pythonw` / `python`，必要时结束进程（先确认再结束）。  
4. 回到本程序点 **刷新**，再点 **连接**。

---

## 3. 如何「精准确认」是谁握着 COM？

Windows 自带的任务管理器**一般不显示**「哪个进程打开了 COM3」。要精准确认，建议：

1. 下载微软 **Sysinternals Process Explorer**（微软官方工具）。  
2. 打开后按 **Ctrl+F**，搜索 **`COM3`**（改成你的口号）。  
3. 搜索结果会指向具体 **进程名 + PID**。

同一工具包里的 **Handle** 命令行也可：

```text
handle64.exe -a COM3
```

（需从 Sysinternals 页面单独下载 handle，与 Process Explorer 同属一套。）

---

## 4. 上线产品里怎么减轻用户困扰？

- 错误弹窗中提示用户：**先拔插 USB**、关 Arduino 监视器，并附上 **本文档链接**。  
- 支持渠道可请用户附上 **任务管理器里含 `daemon_unified` 的进程截图** 或 Process Explorer 的 COM 搜索结果。  
- 若企业环境禁止结束进程，需 IT 放行或统一由服务账号独占串口。

---

## 5. 与 BLE 模式的关系

若环境变量 **`CC_TL_TRANSPORT=ble`**，灯走蓝牙通道，**本页 USB 串口「连接」不适用**。此时「拒绝访问」多来自仍去连 COM 的误操作，请先确认传输模式。
